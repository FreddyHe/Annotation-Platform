## 本次主要修改

1. 训练 loss 显示修复

   之前前端训练日志会出现 `loss: ?`。现在算法服务会在每个 epoch 结束时回传 `train_loss`、`box_loss`、`cls_loss`、`dfl_loss` 等指标，前端会显示真实数值；如果训练刚开始还没采集到 loss，会显示 `loss: 采集中`，不再显示问号。

2. 边端推理改成异步任务

   之前上传图片或 zip 后，前端会一直等待同步接口，训练占 GPU 时容易超时。现在边端模拟会先创建异步 job，再轮询 job 状态。前端可以看到排队、算法推理、写入数据池、VLM 判定、同步 Label Studio、完成或失败等阶段。

3. 边端推理进度信息补充

   现在会显示上传帧数、压缩包解析后的图片数、已处理数量、阶段进度和耗时。缺少图片或 zip 时返回明确的 400 错误，不再包装成 500。

4. GPU 资源互斥

   训练和边端推理共用 GPU 时，现在通过算法服务内的 GPU 锁串行执行，避免训练中边端推理直接抢 GPU 导致长时间卡死或超时。

5. 边端模型缓存

   同一个 `best.pt` 不会在每次边端推理时重复加载，减少单张图片推理时的额外耗时。

6. 训练数据来源记录

   每次训练会记录数据来源摘要、来源任务、训练/验证集数量、标注数量和数据指纹。训练预览中会显示上一轮训练数据，例如当前项目最新完成记录 `#289`：

   ```text
   totalImages: 1638
   trainImages: 1310
   valImages: 328
   totalAnnotations: 1893
   ```

7. 重复训练拦截

   如果没有新增回流数据，系统会阻止用同一批数据再次训练，并提示需要勾选“允许无新增数据时复训同一批数据”。这用于避免用户误以为已经进行了新一轮增量训练。

8. 多轮增量与回滚

   修复后，每次部署新模型或回滚旧模型都会开启新的采集轮次。后续边端回流数据会按 `roundId + deploymentId` 隔离，支持多次增量、对比和回滚。

9. 训练预览旧数据兼容

   修复了旧训练记录 `testResults` 被保存成双层 JSON 字符串时，预览页无法解析 `trainImages/valImages` 的问题。

## 前端测试步骤

### 1. 基础登录与项目进入

1. 打开 `http://localhost:6006/`。
2. 使用 `admin / 123456` 登录平台。
3. 进入项目 `动物检测7`。
4. 确认页面可以正常加载项目详情、训练页、边端模拟页。

预期结果：

- 前端页面正常打开。
- 后端接口不再报连接拒绝。
- 训练状态能看到最新完成记录 `#289`。

### 2. 训练预览测试

1. 进入“模型训练”页面。
2. 打开训练预览或点击开始训练前的确认信息。
3. 核对这些字段：

```text
主项目任务数：1821
已全审增量任务数：7
待跳过未全审增量任务数：200
上一轮训练记录：#289
上一轮训练数据：1310 train / 328 val / 1638 total
```

预期结果：

- 训练预览能显示当前轮次池统计。
- `trainImages/valImages` 不应再显示为 0。
- 如果没有新增已审核数据，系统应提示当前训练数据可能与上一轮一致。

### 3. 重复训练拦截测试

1. 在没有新增回流数据、也没有新审核增量项目的情况下点击“开始训练”。
2. 不勾选“允许无新增数据时复训同一批数据”。

预期结果：

- 系统应阻止训练启动。
- 应提示“本次训练数据与上一次完成训练完全一致，没有检测到新增回流数据”。
- 不应启动新的算法训练任务。

如果确实想强制复训同一批数据，再勾选“允许无新增数据时复训同一批数据”。注意：勾选后会真的启动训练，会占用 GPU。

### 4. loss 显示测试

1. 在确认要训练时启动一次训练。
2. 观察训练日志区域。

预期结果：

- 刚启动时可以短暂显示 `loss: 采集中`。
- epoch 开始推进后，应显示真实 loss 数值，例如：

```text
Epoch 3/25 - loss: 1.2345 · box 0.4567 · cls 0.3456 · dfl 0.4321
```

- 不应再出现 `loss: ?`。

### 5. 重新部署并开启新轮次

1. 进入“边端模拟”页面。
2. 选择最新完成模型记录 `#289`。
3. 点击部署到边端。
4. 部署完成后查看部署列表和当前轮次。

预期结果：

- 旧 active 部署会变为回滚/非 active。
- 新部署会成为 active。
- 系统会创建新的采集轮次，后续回流数据进入新 `roundId`。

这是验证多轮增量的关键步骤。只有重新部署一次修复后的代码，后续数据才会进入新的轮次隔离逻辑。

### 6. 边端模拟视频流推理测试

1. 在“边端模拟”页面选择 active 部署。
2. 上传单张图片或包含图片的 zip。
3. 点击“模拟视频流推理”。
4. 观察进度条和阶段文案。

预期结果：

- 前端应显示上传文件名、文件大小、预计帧数、已用时。
- 进度阶段应依次进入：

```text
QUEUED
SERVER_PROCESSING
PERSISTING
VLM_JUDGING
SYNCING_LABEL_STUDIO
COMPLETED
```

- 如果上传 zip，应能看到解析后的图片数量。
- 如果训练正在占 GPU，边端任务应等待 GPU 资源，不应立即报 read timeout。

### 7. 数据池与 Label Studio 同步测试

边端推理完成后，在边端页面查看池统计：

```text
HIGH
LOW_A_CANDIDATE
LOW_A
LOW_B
DISCARDED
```

预期规则：

- 高置信和 VLM 通过后的 LOW_A 会进入可信池，并同步到主 Label Studio 项目。
- LOW_B 会进入 LOW_B 池，满 100 张后创建增量审核项目。
- 未满批时会显示类似 `LOW_B 未满批等待：78 / 100`。
- DISCARDED 不进入人工审核。

然后进入 Label Studio：

```text
http://localhost:5001/
```

预期结果：

- 主项目 `动物检测7` 中能看到可信数据同步后的任务数量变化。
- LOW_B 满批后，应出现新的 `动物检测7__增量xxx__待审` 项目。
- 增量项目审核完成后，训练预览中的“已全审增量任务数”会增加。

### 8. 回滚测试

1. 在边端模拟页选择历史部署。
2. 点击回滚。
3. 回滚完成后上传少量图片做一次边端推理。

预期结果：

- 系统会创建一个新的 active 部署记录，而不是直接复用旧部署。
- 回滚后的部署会开启新的采集轮次。
- 回滚后产生的数据不会混入上一个模型部署轮次。
