# 增量闭环与新增类别机制设计建议

## 1. 当前问题

现在系统后半段已经形成了“边端部署 -> 边端推理 -> 数据池分流 -> Label Studio 审核 -> 再训练 -> 再部署”的增量闭环，但前半段的两个子 tab 还停留在早期项目视角：

- “数据与标注”更像初始上传和初始 LS 同步入口。
- “结果查看”更像早期检测结果查看入口。
- 增量阶段产生的可信池、LOW_A、LOW_B、已审增量项目、训练数据集、部署轮次，没有自然回流到这两个 tab。

结果是：模型训练已经使用 3k 级别数据，但前面两个 tab 仍然像只知道初始项目，用户会感到“项目状态割裂”。

另一个问题是类别扩展。目前增量链路默认类别集合稳定。如果用户在项目中新增类别，只让边端继续跑旧类别模型是不完整的，因为：

- 旧模型不认识新类别。
- 旧 LS 项目标签配置可能没有新类别。
- 历史图片中可能存在新类别但未标注，直接当负样本训练会污染数据。
- 后续训练、边端部署、结果查看都必须知道当前类别版本。

## 2. 设计原则

建议把项目从“一个静态数据集”升级为“带版本的数据闭环项目”。

核心对象应该统一成以下几类：

- `Project`：项目本体。
- `LabelSchemaVersion`：类别集合版本，例如 v1 只有 `animal`，v2 增加 `cat/dog/person`。
- `IterationRound`：采集/增量轮次。
- `DatasetVersion`：一次训练实际使用的数据快照。
- `ModelTrainingRecord`：一次训练结果。
- `EdgeDeployment`：一次边端部署。
- `InferenceDataPoint`：边端回流数据点。
- `LsSubProject`：Label Studio 审核项目，包括普通 LOW_B 增量和类别扩展增量。

前端所有 tab 都应该围绕这些对象展示，而不是各自维护一套互不相关的视图。

## 3. “数据与标注”应如何联动

建议把“数据与标注”升级为项目数据总览和标注工作台，不只显示初始上传。

它应该至少展示：

- 主 LS 项目任务数。
- 初始上传图片数。
- 已同步到主 LS 的可信数据数。
- LOW_B 等待池数量。
- 已创建的增量审核项目列表。
- 已全审增量项目数量。
- 未全审增量项目数量。
- 当前轮次数据池统计：HIGH、LOW_A_CANDIDATE、LOW_A、LOW_B、DISCARDED。
- 当前训练可用数据量。
- 当前类别版本。
- 各类别标注覆盖情况。

推荐结构：

```text
数据与标注
├── 项目数据总览
│   ├── 初始数据
│   ├── 可信回流数据
│   ├── 增量待审数据
│   └── 已审增量数据
├── Label Studio 同步状态
│   ├── 主项目
│   ├── 增量审核项目
│   └── 同步失败/待补偿数据
├── 类别与标注 schema
│   ├── 当前类别版本
│   ├── 类别列表
│   ├── 新增类别入口
│   └── 类别扩展标注任务
└── 数据筛选视图
    ├── 按轮次
    ├── 按来源
    ├── 按类别
    ├── 按审核状态
    └── 按是否参与训练
```

这样用户在训练前就能理解“这次训练到底会用哪些数据”。

## 4. “结果查看”应如何联动

“结果查看”建议从早期检测结果列表升级为统一结果浏览器。

它应该支持按以下维度筛选：

- 训练记录 ID。
- 部署 ID。
- 采集轮次 roundId。
- 数据来源：初始、可信回流、LOW_B 已审、类别扩展。
- poolType：HIGH、LOW_A、LOW_B、DISCARDED。
- 类别。
- 是否人工审核。
- 是否已参与训练。
- VLM 判定结果。

每张图建议展示：

- 图片。
- 当前模型 prediction。
- LS annotation。
- prediction 与 annotation 差异。
- 置信度。
- 所属轮次。
- 所属部署。
- 是否进入训练数据。
- 如果被丢弃，要显示丢弃原因。

这样“结果查看”就不再是早期结果查看，而是整个主动学习闭环的可解释界面。

## 5. 新增类别的推荐闭环

用户新增类别时，不建议直接把类别加到模型训练里就结束。更合理的是引入“类别 schema 版本”和“类别扩展标注任务”。

### 5.1 用户操作

用户在“数据与标注”中点击“新增类别”，例如新增：

```text
animal -> animal, cat, dog
```

系统创建新的类别版本：

```text
LabelSchemaVersion v1: animal
LabelSchemaVersion v2: animal, cat, dog
```

项目状态进入：

```text
待进行新增类别标注
```

前端提示：

```text
已新增类别 cat、dog。当前历史数据未覆盖这些类别，请创建类别增量标注任务。
```

### 5.2 生成类别扩展标注任务

新增类别后，系统应支持两种方式生成标注任务：

1. 用户主动上传新类别样本。
2. 系统从已有数据和边端回流数据中主动挖掘疑似新类别样本。

主动挖掘可以使用：

- 低置信 LOW_A/LOW_B 样本。
- VLM 判断可能包含新类别的样本。
- DINO/开放词汇检测按新类别文本提示召回候选图。
- 旧模型误检或漏检较多的样本。

这些任务进入一个特殊 LS 子项目：

```text
动物检测7__类别增量_v2__待审
```

它和普通 LOW_B 增量项目不同，核心目标不是纠正旧类别，而是补齐新类别标注。

### 5.3 历史数据如何处理

新增类别最大的风险是历史图片中可能存在新类别但没有标注。如果直接把这些图片作为“不含新类别”的负样本训练，会造成模型学习错误。

建议做法：

- 历史图片默认标记为 `schema v1`。
- 新类别训练时，只有经过 v2 审核的图片才作为 v2 完整标注样本。
- 未经过 v2 审核的历史图片不要强行当作新类别负样本。
- 可以抽样或主动召回历史图片做 v2 补标。

也就是说，训练集需要记录每张图的 schema 覆盖情况：

```text
image_001.jpg: schema v1 reviewed
image_002.jpg: schema v2 reviewed
```

模型训练 v2 时，应该优先使用 v2 reviewed 数据。v1 数据可以继续用于旧类别，但要避免错误地惩罚新类别。

### 5.4 Label Studio 同步

新增类别后，需要同步 LS label config：

- 主 LS 项目增加新类别。
- 后续增量审核项目使用新 label config。
- 已存在的待审增量项目需要标记其 schema 版本。

不建议静默改旧项目里的历史任务语义。更稳妥的是：

- 旧任务保留原 annotation。
- 新类别标注通过新的类别增量项目完成。
- 必要时提供“迁移/补标旧任务”的显式按钮。

### 5.5 训练逻辑

训练时应明确使用哪个类别版本：

```text
训练记录 #400
labelSchemaVersion: v2
classes: animal, cat, dog
datasetVersion: dataset_v2_round_326
```

训练数据构建规则：

- 包含主项目中已同步且 schema 覆盖足够的数据。
- 包含已全审普通增量项目。
- 包含已全审类别增量项目。
- 排除未全审项目。
- 排除 schema 不兼容的数据，或降级为只参与旧类别训练。

训练完成后，模型记录必须绑定类别版本。

### 5.6 边端部署逻辑

边端部署也要绑定类别版本：

```text
deploymentId: 220
modelRecordId: 400
labelSchemaVersion: v2
classes: animal, cat, dog
```

边端推理输出必须使用当前模型类别映射。数据池分流也应按类别记录：

- 每个检测框的类别。
- 每个类别的平均置信度。
- 每张图的整体池类型。
- 是否包含新增类别。

这样后续 HIGH、LOW_A、LOW_B 不再只是单类别动物检测，而是多类别感知的增量池。

## 6. 回滚与类别版本的关系

回滚时必须考虑类别版本兼容。

例如：

- 当前部署是 v2：animal、cat、dog。
- 用户回滚到 v1：animal。

系统应提示：

```text
目标模型只支持 schema v1，不包含 cat、dog。回滚后边端将不再识别新增类别。
```

回滚可以允许，但要明确：

- 新轮次使用回滚模型的类别版本。
- v2 数据不应丢失。
- 后续再切回 v2 模型时，v2 数据仍可继续使用。

## 7. 推荐的项目闭环

完整闭环建议如下：

```text
1. 初始数据上传
2. 同步主 Label Studio 项目
3. 初始人工审核或确认 prediction
4. 初始训练
5. 部署到边端
6. 边端推理产生回流数据
7. HIGH/LOW_A 可信数据入主 LS
8. LOW_B 满批进入增量审核 LS 项目
9. 人工审核增量项目
10. 训练预览展示本次可用数据
11. 训练新模型
12. 新模型部署开启新轮次
13. 重复 6-12
```

新增类别时插入一条分支：

```text
A. 用户新增类别
B. 创建 LabelSchemaVersion
C. 提示需要类别增量标注
D. 创建类别扩展 LS 项目
E. 审核新类别样本
F. 将类别增量数据纳入 DatasetVersion
G. 训练多类别模型
H. 部署多类别模型
I. 后续边端回流按新类别版本继续闭环
```

## 8. 我建议的实现优先级

### 第一阶段：打通现有单类别增量的前端闭环

目标是让用户在“数据与标注”和“结果查看”中看见后半段增量数据。

应做：

- 数据与标注展示主项目、增量项目、数据池、训练可用数据。
- 结果查看支持按 round/deployment/poolType/filter 查询边端回流结果。
- 每张图展示是否已同步 LS、是否已审核、是否参与训练。

这一阶段不改变训练算法，只改变项目视图和数据解释。

### 第二阶段：引入类别版本

应做：

- 新增 `LabelSchemaVersion` 概念。
- 训练记录绑定类别版本。
- 部署记录绑定类别版本。
- LS 子项目绑定类别版本。
- 数据点绑定类别版本。

这一阶段先把数据结构打稳。

### 第三阶段：新增类别标注闭环

应做：

- 数据与标注中支持新增类别。
- 新增类别后创建类别扩展任务。
- 类别扩展任务进入 LS。
- 已审类别扩展任务进入训练数据。
- 训练多类别模型。

### 第四阶段：主动挖掘新类别样本

应做：

- 用 VLM/DINO/低置信样本挖掘疑似新类别。
- 支持候选样本确认。
- 支持按类别查看覆盖率。

## 9. 结论

你提出的“新增类别后，在数据与标注提示用户进行增量类别标注，再把这部分标注以增量形式加入项目和 LS，后续训练与边端模拟基于新的类别数量运行”是合理的。

但需要补充一个关键前提：必须引入类别 schema 版本。否则新增类别后，旧数据、旧 LS 项目、旧模型、新模型之间会混在一起，后续训练和回滚都会变得不可解释。

推荐方向是：

```text
数据闭环按 round/version 管理；
类别闭环按 labelSchemaVersion 管理；
训练闭环按 datasetVersion 管理；
部署闭环按 modelRecordId + labelSchemaVersion 管理。
```

这样项目才能从单次增量升级为可多轮、可扩类、可回滚、可解释的完整主动学习系统。

## 10. 开发前需要确认和讨论的问题

下面这些问题建议在正式开发前确认。它们会直接影响数据库结构、前端交互、训练数据构建、Label Studio 同步和回滚逻辑。

### 10.1 第一阶段范围：先做哪一层闭环

建议先确认第一版开发范围。

可选范围：

```text
A. 只改前端展示：数据与标注、结果查看接入现有增量数据。
B. 前端展示 + 后端查询接口：增加按 round/deployment/pool/schema 的统一查询。
C. 同时引入类别版本和 DatasetVersion，为新增类别做结构准备。
D. 一次性做完多类别新增、类别增量 LS 项目、训练和部署联动。
```

我的建议是先做 `B + 一部分 C`：

- 先让“数据与标注”和“结果查看”真正看到增量链路。
- 同时把接口返回结构设计成支持 `labelSchemaVersion`，但新增类别的完整闭环可以第二步做。
- 不建议第一版直接做 `D`，因为它会同时改数据模型、LS 同步、训练转换、边端推理、前端多个页面，风险较大。

需要你确认：

```text
第一版是否接受分阶段交付？
第一版是否必须包含“新增类别”功能，还是先完成单类别增量闭环联动？
```

### 10.2 “数据与标注”子 tab 的产品定位

需要确认“数据与标注”未来到底承担哪个角色。

方案 A：项目数据总览

- 展示初始数据、主 LS、可信池、LOW_B、增量项目、训练可用数据。
- 更偏管理视角。
- 不直接做逐图审核，只提供跳转 LS。

方案 B：平台内轻量标注工作台

- 除了总览，还在平台内展示图片和框。
- 可以做一些轻量确认、筛选、同步。
- 复杂度明显更高，容易和 LS 功能重复。

我的建议：

```text
第一版按方案 A 做，保持 LS 作为人工标注主入口。
平台负责聚合状态、解释数据来源、触发同步和训练。
```

需要你确认：

```text
你希望“数据与标注”只是总览和跳转，还是也要能在平台内直接改标注？
```

### 10.3 “结果查看”子 tab 的产品定位

需要确认“结果查看”展示什么结果。

可展示对象有三类：

```text
1. 初始自动检测结果。
2. 边端回流推理结果。
3. 已进入训练数据的 LS annotation / prediction 对比结果。
```

我的建议是改成“统一结果浏览器”，默认展示边端回流和训练数据来源。

第一版推荐筛选项：

- 轮次 `roundId`
- 部署 `deploymentId`
- 数据池 `poolType`
- LS 同步状态
- 人工审核状态
- 是否参与训练
- 置信度区间
- 文件名/任务 ID

需要你确认：

```text
结果查看默认应该显示“最新轮次结果”，还是显示“全部项目结果”？
是否需要保留原来的初始检测结果视图？
```

### 10.4 是否引入 DatasetVersion

当前 `ModelTrainingRecord` 里已经保存了一部分训练数据元信息，但如果要做严谨闭环，建议增加独立的 `DatasetVersion` 或类似实体。

它用于记录：

- 本次训练用了哪些 LS 项目。
- 用了哪些任务 ID。
- 用了哪些边端数据点 ID。
- 对应哪个 `roundId`。
- 对应哪个 `labelSchemaVersion`。
- 数据集 fingerprint。
- train/val/test 划分。
- 是否包含类别扩展数据。

优点：

- 训练数据可追溯。
- 支持多次增量。
- 支持回滚后比较不同模型。
- 支持“本次训练和上次训练是不是同一批数据”的精确判断。

缺点：

- 需要新增表和迁移旧记录。
- 前端训练历史要适配新结构。

需要你确认：

```text
是否允许新增 DatasetVersion 相关表？
旧训练记录是否只做兼容展示，不强行迁移完整来源明细？
```

### 10.5 是否引入 LabelSchemaVersion

如果要支持新增类别，我认为必须引入 `LabelSchemaVersion`。

需要确认的规则：

```text
1. 类别只能新增，还是允许重命名/删除/合并？
2. 类别顺序是否固定？因为 YOLO class_id 依赖顺序。
3. 旧模型的类别版本如何展示？
4. 回滚到旧类别模型时，是否允许？
5. 新增类别后，旧 LS 项目是否更新 label config？
```

我的建议：

- 第一版只允许新增类别。
- 不允许删除类别。
- 重命名要作为高风险操作，先不做或只改 displayName，不改训练 class key。
- 每个类别有稳定 `classKey`，显示名可以变化。
- YOLO class_id 按 schema version 固化。

需要你确认：

```text
新增类别第一版是否只支持“新增”，不支持删除/合并/重排？
类别是否需要中英文名，还是只需要一个 label？
```

### 10.6 新增类别后，历史数据如何处理

这是新增类别里最重要的问题。

例如旧项目只有 `animal`，后来新增 `cat`。历史图片里可能有猫，但旧标注没有猫。如果直接拿历史图片训练 cat，可能会把猫当背景，污染训练。

可选策略：

方案 A：严格策略

- 只有经过新 schema 审核的图片才参与新类别训练。
- 最安全，但新增类别初期数据少。

方案 B：兼容策略

- 旧 schema 数据继续训练旧类别。
- 新类别只从新审核数据中学习。
- 需要训练转换逻辑能区分“旧类别监督”和“新类别监督”。

方案 C：粗放策略

- 更新类别后所有历史图都按新 schema 训练。
- 实现简单，但新类别容易被错误当背景。

我的建议：

```text
采用方案 B。
旧数据继续保留价值，但新类别必须来自 v2 审核数据。
```

需要你确认：

```text
你是否接受新增类别后，需要先做一批“类别增量标注”，否则新类别不能进入训练？
```

### 10.7 类别增量 LS 项目如何创建

新增类别后，类别增量任务来源需要确认。

可选来源：

```text
1. 用户上传新类别样本。
2. 从现有主项目抽样补标。
3. 从 LOW_A/LOW_B/DISCARDED 里筛选疑似新类别。
4. 用 DINO/VLM 按新类别文本召回候选图。
```

我的建议第一版支持：

- 用户上传新类别样本。
- 从现有数据池手动选择样本。
- DINO/VLM 主动召回作为第二版增强。

需要你确认：

```text
新增类别第一版是否必须支持自动挖掘候选图？
还是先支持手动上传/手动选择？
```

### 10.8 主 LS 项目和增量 LS 项目的关系

现在可信数据会同步到主 LS，LOW_B 满批进入增量项目。

新增类别后需要确认：

```text
1. 主 LS 项目是否立刻更新 label config？
2. 已存在的未审增量项目是否更新 label config？
3. 旧的已审增量项目是否保持原 schema？
4. 类别增量项目审核完成后，是否合并回主 LS？
```

我的建议：

- 主 LS 项目更新到最新 schema。
- 新创建的增量项目使用最新 schema。
- 旧的已审项目保持原 schema，只记录它属于旧 schema。
- 旧的待审项目可以提示“schema 已过期”，由用户选择继续按旧 schema 审，或重新生成新 schema 增量项目。

需要你确认：

```text
旧待审增量项目遇到新增类别时，是继续保留，还是强制作废并重建？
```

### 10.9 训练数据构建规则

需要确认训练时如何选择数据。

当前逻辑大致是：

```text
主 LS 项目 + 已全审增量项目
```

升级后建议变为：

```text
选择 labelSchemaVersion
选择 datasetVersion 构建策略
包含主 LS 已审核/可信数据
包含已全审普通增量项目
包含已全审类别增量项目
排除未全审项目
排除 schema 不兼容数据或降级使用
```

需要你确认：

```text
训练是否永远使用“最新类别版本”？
是否允许用户选择用旧类别版本训练/复训？
```

我的建议：

- 默认使用最新类别版本。
- 高级选项中允许选择旧 schema 复训，用于回滚和对比。

### 10.10 边端数据池多类别分流规则

单类别时，一张图的平均置信度足够决定 HIGH/LOW_A/LOW_B/DISCARDED。

多类别后需要确认：

```text
1. 按图片整体平均置信度分流？
2. 按每个类别分别分流？
3. 只要包含低置信框，就进入 LOW_B？
4. 新类别是否更保守，优先进入人工审核？
```

我的建议：

- 图片有整体 poolType。
- 每个检测框保留 classId/className/confidence。
- 如果包含新增类别，分流阈值更保守。
- 新类别早期尽量进入 LOW_A/LOW_B 审核，而不是直接 HIGH 入主项目。

需要你确认：

```text
新增类别早期是否要降低自动入主 LS 的比例，更多进入人工审核？
```

### 10.11 VLM 在新增类别中的角色

VLM 可以用于：

- 判断低置信框是否可信。
- 判断图片是否可能包含新增类别。
- 给出候选类别理由。

需要确认：

```text
VLM 是否作为强规则，能直接决定 LOW_A/LOW_B？
还是只作为辅助解释，由人工最终决定？
```

我的建议：

- 普通旧类别：VLM 可以继续辅助分流。
- 新增类别早期：VLM 只做候选召回和解释，不直接作为最终标注。

### 10.12 前端交互入口

需要确认新增类别入口放在哪里。

推荐位置：

```text
数据与标注 -> 类别与标注 schema -> 新增类别
```

新增类别后显示：

```text
当前类别版本：v2
新增类别：cat, dog
状态：需要类别增量标注
按钮：创建类别增量标注项目
按钮：上传新类别样本
按钮：从数据池选择候选样本
```

需要你确认：

```text
新增类别是否只允许项目管理员操作？
是否需要新增类别审批流程？
```

### 10.13 回滚交互

回滚不仅是模型回滚，还涉及类别版本。

需要确认：

```text
1. 回滚到旧模型时，是否允许类别数量减少？
2. 如果减少，前端是否强提示？
3. 回滚后新采集轮次使用旧 schema 还是项目最新 schema？
```

我的建议：

- 允许回滚。
- 回滚部署绑定目标模型的 schema。
- 明确提示“回滚后不会识别新增类别”。
- 项目最新 schema 不降级，只是当前部署模型 schema 降级。

### 10.14 旧数据兼容和迁移

当前已有数据中，一些记录可能没有：

- `labelSchemaVersion`
- `datasetVersion`
- 完整 sourceItems
- 每框类别详情

需要确认：

```text
旧数据是否全部标记为 schema v1？
旧训练记录是否补 datasetVersion？
旧边端数据是否需要迁移？
```

我的建议：

- 旧项目默认生成 `LabelSchemaVersion v1`。
- 旧训练记录只做兼容展示，不做强迁移。
- 新训练开始后生成完整 DatasetVersion。
- 旧边端数据按能推断的方式补 `roundId/schemaVersion`，不能推断的只读展示。

### 10.15 是否需要权限和审计

新增类别、作废增量项目、回滚模型都会影响项目语义。

需要确认：

```text
是否需要记录操作人、操作时间、操作原因？
是否需要限制只有管理员能新增类别/回滚/作废项目？
```

我的建议：

- 至少记录审计日志。
- 新增类别、回滚、作废增量项目限制为项目 owner/admin。

### 10.16 验收标准

建议提前约定验收标准。

第一阶段验收可以是：

```text
1. 数据与标注能看到初始数据、主 LS、可信池、LOW_B、增量项目、训练可用数据。
2. 结果查看能按 round/deployment/poolType 查看边端回流结果。
3. 每张结果能看到是否同步 LS、是否人工审核、是否进入训练。
4. 训练预览和数据与标注显示的数据口径一致。
5. 训练历史能看到本次 DatasetVersion 或数据来源摘要。
```

新增类别阶段验收可以是：

```text
1. 可以新增类别并生成新 LabelSchemaVersion。
2. 系统提示需要类别增量标注。
3. 可以创建类别增量 LS 项目。
4. 类别增量项目审核完成后可进入训练。
5. 新训练模型类别数量正确。
6. 边端部署使用新类别模型并输出新类别预测。
7. 回滚旧模型时有类别版本提示。
```

### 10.17 我建议你优先拍板的 6 个问题

为了进入开发，最需要你先确认的是：

```text
1. 第一版是否先做“数据与标注/结果查看联动”，新增类别放第二阶段？
2. 是否允许新增 DatasetVersion 表？
3. 是否允许新增 LabelSchemaVersion 表？
4. 新增类别第一版是否只支持新增，不支持删除/合并/重排？
5. 类别增量样本第一版是否先支持手动上传/手动选择，不做自动挖掘？
6. 旧待审增量项目遇到新增类别时，是保留旧 schema，还是作废重建？
```

确认这 6 点后，就可以把开发拆成可执行任务。
